## 二、核心算法

### 2.1 目标检测：YOLOv8

- **模型**：`yolov8n.pt`（Nano 版，推理速度优先）
- **部署方式**：Ultralytics Python API，ROS 2 节点内直接调用
- **输入**：仿真相机图像（`sensor_msgs/Image`），先节流至 5 Hz（参数可改） 再送入推理，这里做了缓冲区设计
- **输出**：带标注框的图像 + 检测结果 JSON

**推理流程**（`yolo_detector.py`）：

```
相机话题 /camera/image_raw
       ↓ 节流节点（5 Hz，可调）
/camera/image_throttled
       ↓ ROS 消息 → NumPy RGB
       ↓ YOLO.predict()
       ↓ 结果解析（类别 / 置信度 / 边界框）
       ↓ 标注渲染（PIL + HUD 叠加）
/camera/annotated（发布）+ 结果 JSON 存档
```

**实时性能指标**（`PerceptionMetrics`）：
- 滚动窗口 FPS（输入帧率 + 推理帧率）
- 推理延迟 P50 / P95（毫秒）
- 各类别检测总计数

### 2.2 仿真场景：车辆巡逻动画

- **脚本**：`sim/launch/animate_vehicles.py`
- **机制**：每 0.5 秒通过 Gazebo `set_pose` 服务更新车辆位置，模拟直线来回运动

**巡逻路线**（三辆动态车辆）：

| 车辆    | 路段方向 | 范围                   | 速度  |
| ------- | -------- | ---------------------- | ----- |
| `car_0` | 东西向   | x: −80 → −20，y = 76.2 | 8 m/s |
| `car_3` | 南北向   | x = 2.8，y: −40 → 20   | 7 m/s |
| `car_5` | 南北向   | x = −12.1，y: 0 → 60   | 6 m/s |

**朝向计算**：端点连线方向取反正切，掉头时自动加 π，通过四元数传给 Gazebo：

```python
yaw = atan2(y2 - y1, x2 - x1)   # 前进方向
qz  = sin(yaw / 2)
qw  = cos(yaw / 2)
```

## 三、遇到的问题与解决方法

### 3.1 仿真场景搭建阶段

| 问题                       | 原因                                                     | 解决方法                                                                      |
| -------------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 车辆调用 `set_pose` 无响应 | 模型设置了 `<static>true</static>`，物理引擎跳过位置计算 | 车辆去掉 `static` 标签，仅行人保留                                            |
| `set_pose` 请求格式错误    | 误用了嵌套 `pose: {}` 包装，不符合 `gz.msgs.Pose` 协议   | 改为扁平格式：`name, position{}, orientation{}`                               |
| 坐标偏差大（路面找不到）   | baylands park 地图中心在 `(205, 155)`，不是原点          | 实测扫描确认：NS 路段 x ≈ 0，EW 路段 y ≈ 76                                   |
| 仿真 CPU 负载过高          | 14 个动态模型 @ 250 Hz 物理更新率                        | 降低更新率至 100 Hz，负载降至可接受范围                                       |
| 重载仿真后车辆朝向错误     | SDF 中 yaw 值为估算值（1.5708），与手动摆放时不符        | 从 `/world/baylands_2026cv/pose/info` 话题读取真实四元数，换算 yaw 后写回 SDF |

### 3.2 感知流水线阶段

| 问题                     | 原因                                        | 解决方法                                                       |
| ------------------------ | ------------------------------------------- | -------------------------------------------------------------- |
| YOLO 节点 import 失败    | 虚拟机无 GPU，`ultralytics` 版本要求不一致  | 用 `try/except` 包裹导入，CPU 模式降级运行                     |
| 图像话题高频导致推理堆积 | Gazebo 相机默认 30 Hz，超出推理能力         | 增加节流节点，降至 5 Hz 送入 YOLO                              |
| 检测结果无法可视化       | ROS 2 中 `sensor_msgs/Image` 格式需手动转换 | 封装 `image_message_to_rgb` 和 `rgb_to_image_message` 工具函数 |

### 3.3 系统卡死问题（重点记录）

**现象**：完整链路运行（PX4 + Gazebo + ROS 2 + 可视化）一段时间后系统逐步变卡，最终接近卡死；不起飞也会随时间加重，停止流程后明显恢复。

**诊断过程**：

1. 初步排查排除了内核硬错误，确认是**内存与 swap 压力持续累积**
2. 确认 Gazebo 已启用 GPU 渲染，但 RAM 仍线性增长——GPU 只加速渲染，不能消除主机内存中的数据传输积压
3. 连续采样各进程 RSS，定位到主增长源在 **Gazebo server 进程**，而非 YOLO 推理侧

**根因**：相机源头帧率/分辨率较高，Gazebo 内部与 ROS 2 桥接链路存在无界缓冲，后段消费速率长期低于前段生产速率，导致 RAM 线性增长。

**落地的解决措施**：

| 措施             | 具体操作                                                                | 效果                         |
| ---------------- | ----------------------------------------------------------------------- | ---------------------------- |
| 链路有界化       | ROS 2 中插入 `image_throttle` 节点（只保留最新帧，丢弃旧帧，5 Hz 重发） | 后段内存由无界增长变为有界   |
| 源头降流         | 自定义相机模型，分辨率 640×360，帧率 10 Hz                              | 减少 Gazebo 内部缓存生成速率 |
| 自定义仿真资源层 | 用户目录建立 `sim/custom_gz/`，不修改官方文件                           | 避免版本更新破坏配置         |
| 运行保护         | `view_annotated.sh` 增加内存阈值守护（可用内存低于阈值时自动停止）      | 防止最终卡死                 |

**验证方法**（调参后执行）：
```bash
# 检查相机实际帧率
timeout 5 ros2 topic hz /camera/image_raw
# 连续观测关键进程 RSS
ps -o pid,rss,comm -C ruby,parameter_bridge,image_throttle,yolo_detector
```

### 3.4 朝向读取难点（重点记录）

**问题**：尝试多种方式读取 Gazebo 模型当前朝向均失败：
- `grep` 正则无法区分 `position.z` 和 `orientation.z`（同名字段）
- `gz model --pose` 命令不存在（Gazebo Harmonic 已弃用）
- `dynamic_pose/info` 话题对静止模型无输出

**最终解决**：
1. 将 `pose/info` 话题原始输出重定向到文件（`> /tmp/poses.txt`）
2. 用 `sed` 按行号精确截取每辆车的 15 行数据块
3. 提取 `orientation.z`（即 qz）和 `orientation.w`（即 qw）
4. 公式：`yaw = atan2(2 × qw × qz, 1 − 2 × qz²)`
